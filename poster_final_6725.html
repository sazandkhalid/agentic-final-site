Self-Constructing Agents with Verified Skill Persistence

Procedural Memory in Code-Acting Language Agents through a Four-Gate Verification Pipeline
Ihsan Alaeddin · Sazan Khalid
DEPARTMENT OF DATA SCIENCE AND ANALYTICS · DSAN 6725 · GEORGETOWN UNIVERSITY
Introduction
The integration knowledge problem

Most language-model agents are stateless. Every new task starts at zero, and engineering knowledge produced during one session evaporates when it ends. In domains where the same problems recur, a stateless agent is structurally incapable of getting better at the work.

We present an architecture in which an agent writes, verifies, and accumulates a permanent library of executable Python skills as a by-product of normal task execution — and reuses them across future sessions through composition, evolution, and decay.

The motivating domain is the November 2025 SWIFT ISO 20022 migration, which retired the legacy MT format and forced 11,000+ banks to translate every cross-border payment into a new structured XML standard.

Research Questions
RQ1 · Performance
Does an agent with an accumulated verified skill library perform measurably better on novel tasks than one starting cold?
RQ2 · Mechanism
Where does the benefit come from — direct skill reuse, accumulated context quality, episodic memory, or some combination?
RQ3 · Verification
Is the four-gate verification pipeline strict enough to keep the library trustworthy without suppressing skill formation entirely?
Related Work

ReAct (Yao et al., 2022) established interleaving reasoning with structured tool calls, but kept the action space fixed at design time.

CodeAct (Wang et al., 2024) demonstrated that emitting Python directly is more expressive than JSON tool calls, reducing model calls ~30% on multi-step tasks.

Voyager (Wang et al., 2023) showed open-ended skill accumulation in Minecraft. We extend this paradigm with verified persistence, lifecycle management, and a financial-domain application where correctness is non-negotiable.

Limitations
01 Sample size. n=6 benchmark tasks; single run — directional, not statistically significant.
02 Single model. Agent, validation, and judge all use the same model — correlated failure modes.
03 Domain mismatch. Payment-domain exploration; FastAPI benchmark — isolates context-quality benefit.
04 No ablations. The three benefit channels are not independently tested.

Skill Lifecycle
Four mechanics that keep the library accurate and lean

Composition. Before generating new code, the orchestrator retrieves the three highest-similarity skills and asks the model whether they can be combined to solve the task. If verified, the composition itself becomes a saved skill.

Evolution. When a skill's fail_count reaches 2 and exceeds success_count, the model is asked to diagnose and patch it. The patched version is verified through the full pipeline; the original is archived as _v1.py.

Decay. Each skill carries a relevance score: success / (1 + days_unused × 0.1). Below 0.5 with age > 7 days the skill is de-prioritised; below 0.1 with age > 14 days it is auto-archived.

Failure recovery. A classifier categorises low-graded outputs into five types — hallucination, missing_context, logical_inconsistency, tool_misuse, stylistic_misalignment — and applies a type-specific recovery prompt before marking the task failed.

Methods
Cold–Warm within-subject design

Cold: wipe skills + episodes → run benchmark.
Warm: wipe → run 20 exploration tasks → run same benchmark. Identical model, prompt scaffolding, retrieval, and verification gates — only the library state varies.

Benchmark: 6 tasks (2 per tier) against the FastAPI codebase. Tier 1 structural retrieval · Tier 2 pattern replication · Tier 3 novel synthesis.

Scoring: 0–3 per task. LLM-as-judge (0–2) plus bonuses for code execution, real file references, and verified-skill use. Capped at 3 to avoid double-counting.

Findings
The multi-channel benefit

Tier 1 produced the largest gain (+1.00) despite being the tier least amenable to direct skill reuse. The benefit arrives through three channels:

CHANNEL 1
Episodic memory
Twenty exploration episodes injected as context; missing_context failures dropped 3 → 2 cold-to-warm.
CHANNEL 2
Tool-call discipline
Non-empty skill manifest implicitly demonstrates the read-files-first pattern; warm tasks called read_file more reliably.
CHANNEL 3
Direct skill reuse
The channel the architecture was designed for did the least work in this run.
